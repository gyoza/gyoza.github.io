Prometheus Metric Cardinality Calculator

Test your metric design before it breaks production

Add each label and how many unique values it can have

How to find unique label values:

For existing metrics in Prometheus/GMP, run:

count(count by (label_name) (your_metric_name))

Examples:

  • count(count by (status) (http_requests_total))
  • count(count by (environment) (deployments_total))

For new metrics, count the total possible values for each label:

  • orgId: If you use orgId as a label, set the count to the total number of organizations in your system (e.g., 500 orgs = 500)
  • userId: If you use userId, set it to total users (e.g., 10,000 users = 10,000) - ⚠️ This will create high cardinality!
  • environment: Count your environments (e.g., dev, staging, prod = 3)
  • region: Count deployment regions (e.g., us-west, us-east, eu-central = 3)
  • status: Count possible status values (e.g., success, failure, timeout = 3)
  • pod_name: If you label by pod name, count max pods (e.g., 20 pods in your deployment = 20)
  • accountId: Total accounts in your platform (e.g., 2,000 accounts = 2,000) - ⚠️ High cardinality!

Remember: Labels with IDs (orgId, userId, accountId, requestId) multiply together - be very careful!

Cardinality Guidelines

  • Low (< 1,000): Safe - Great for operational metrics
  • Medium (1,000 - 10,000): Caution - Monitor closely, may impact performance
  • High (> 10,000): Danger - Will likely cause scrape failures, OOM, slow queries

Pro Tip: Avoid labels like accountId, pipelineId, userId, requestId - they create unbounded cardinality.
